# Reply handler — обработка входящего ответа клиента

Подставляется как `user`-сообщение после system prompt.
Используется всегда, когда от лида приходит новое incoming-письмо.

---

- Компания: **{company_name}**, {city}

- Текущий статус в воронке: **{lead_status}**
- Документы, которые мы УЖЕ отправляли этому лиду: {sent_documents}

## ВСЯ ИСТОРИЯ ПЕРЕПИСКИ (в хронологии)
{conversation_history}

## ПОСЛЕДНЕЕ ВХОДЯЩЕЕ СООБЩЕНИЕ (на него отвечаем)
{incoming_message}

## ЦЕЛЬ ОТВЕТА
Каждым письмом двигай к передаче менеджеру: добывай контакты ЛПР (ФИО, должность,
email, телефон — чего ещё нет) и поднимай интерес. Деликатно и кратко, одно
предложение/вопрос за раз, без давления.

## ЗАДАЧА
1. **Классифицируй намерение** клиента (поле `intent`):
   - `interest` — интерес, задаёт вопросы по продукту, просит материалы
   - `question` — конкретный вопрос без явного интереса. СЮДА ЖЕ относится
     **дефлекшн** «передам ответственному / это не ко мне / перешлю специалисту»
     БЕЗ контакта — это НЕ автоответчик: отвечаем и вежливо просим контакт ЛПР.
   - `objection` — возражение (цена, уже есть поставщик, нет спроса, «подумаем»)
   - `ready` — ЛЮБОЙ сигнал готовности: «давайте работать», «готовы заказать»,
     «выставите счёт/КП», просит договор/реквизиты/оплату, называет объёмы →
     передаём менеджеру (`handed_to_manager`, не дожимаем сами)
   - `reject` — отказ, «не интересно», «не работаем с этим»
   - `unsubscribe` — просит не писать больше
   - `autoreply` — ТОЛЬКО настоящий бот-автоответ (out-of-office, отпуск,
     системное уведомление). Живое «передам коллеге» — это `question`, НЕ autoreply.
   - `out_of_scope` — юрпретензии, индивидуальные скидки, откаты (стоп-слова
     из `objections.md`)

2. **Определи новый статус воронки** (`new_status`) — допустимы только три значения:
   - `interest` → `in_dialog` (продолжаем переписку)
   - `question` → `in_dialog`
   - `objection` → `in_dialog` (продолжаем диалог, отрабатываем возражение)
   - `ready` → `handed_to_manager` (готов к сделке — передаём менеджеру)
   - `reject` → `lost`
   - `unsubscribe` → `lost`
   - `autoreply` → `in_dialog` (статус не важен: при autoreply ответ не шлём)
   - `out_of_scope` → `handed_to_manager` (передаём менеджеру в любом случае)

3. **Сгенерируй ответное письмо** (`reply_text`), соблюдая правила system
   prompt'а:
   - 2–3 коротких абзаца, деловой деликатный тон, без воды/канцелярита/эмодзи
   - ответить по существу на то, что спросил клиент
   - **если контакт ЛПР ещё неизвестен** — добавь одно вежливое предложение узнать,
     к кому обращаться по закупкам (имя/должность + email или телефон). Не дави.
   - на вопрос о продукте — ответь по базе и, к месту, подай **FITSIZ.APP** (наш козырь)
   - если уместно — предложи ОДИН документ (прайс-лист или презентацию)
   - на возражение — согласись с чувством, дай факт, предложи маленький шаг;
     не сдавайся после первого «нет», но и не напирай
   - не повторяй подпись
   - для `reject` / `unsubscribe` — короткое вежливое подтверждение, без
     «может передумаете»
   - для `ready` / `out_of_scope` (передача менеджеру) — РОВНО одно сообщение
     по шаблону передачи из system prompt:
     «Отлично. Передаю вас старшему менеджеру — <имя менеджера> свяжется с
     вами в ближайшее время и поможет с заказом.» Имя менеджера бери из
     system prompt. Условия сделки, счёт, договор, цену сам НЕ обсуждай.
   - для `autoreply` (настоящий бот) — reply_text оставь пустой, `should_send = false`

4. **Subject** (`reply_subject`) — либо «Re: {исходная тема}», либо
   новая короткая тема если контекст сильно сменился.

5. **Вложения** (`attachments`) — РОВНО два допустимых файла, других нет:
   - `pricelist_fitsiz_2026.xlsx` — оптовый прайс-лист. Когда спрашивают про
     цены, опт, скидки, стоимость, условия закупки.
   - `presentation_fitsiz.pdf` — презентация FITSIZ. Когда хотят узнать о
     компании, продукции, ассортименте, при первом знакомстве.

   Каталога, КП, сертификата, листовок как файлов НЕ существует — не указывай их.
   НЕ прикладывай документ, который уже был отправлен этому лиду (см.
   `sent_documents` выше). Если нужно повторно — поясни это в `reply_text`.

6. **`is_warm`** — булево: `true` если это клиент, которому нужно передать
   менеджера (`new_status = handed_to_manager`).

7. **`should_send`** — булево: `false` для автоответа или когда агент должен
   промолчать; `true` в остальных случаях.
