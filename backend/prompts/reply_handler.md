# Reply handler — обработка входящего ответа клиента

Подставляется как `user`-сообщение после system prompt.
Используется всегда, когда от лида приходит новое incoming-письмо.

---

## КОНТЕКСТ
- Компания: **{company_name}**, {city}
- Контактное лицо: {contact_name}
- Текущий статус в воронке: **{lead_status}**
- Документы, которые мы УЖЕ отправляли этому лиду: {sent_documents}

## ВСЯ ИСТОРИЯ ПЕРЕПИСКИ (в хронологии)
{conversation_history}

## ПОСЛЕДНЕЕ ВХОДЯЩЕЕ СООБЩЕНИЕ (на него отвечаем)
{incoming_message}

---

## ЗАДАЧА
1. **Классифицируй намерение** клиента (поле `intent`):
   - `interest` — интерес, задаёт вопросы по продукту, просит материалы
   - `question` — конкретный вопрос без явного интереса
   - `objection` — возражение (цена, уже есть другой поставщик, нет спроса)
   - `ready` — готов работать, просит договор/реквизиты, «давайте
     оформлять» → это `warm`, передаём менеджеру
   - `reject` — отказ, «не интересно», «работаем с другими и не
     рассматриваем»
   - `unsubscribe` — просит исключить, не писать больше
   - `autoreply` — автоответчик, отпуск, уведомление-пересылка
   - `out_of_scope` — юрпретензии, индивидуальные скидки, откаты — стоп-слова
     из `objections.md`

2. **Определи новый статус воронки** (`new_status`):
   - `interest` → `interested` (или `negotiating`, если уже обсуждают объёмы/доставку)
   - `question` → `replied` (если был `new`/`contacted`), иначе сохранить текущий
   - `objection` → `replied` (продолжаем диалог)
   - `ready` → `warm`
   - `reject` → `rejected`
   - `unsubscribe` → `unsubscribed`
   - `autoreply` → не меняем статус (оставляем как есть)
   - `out_of_scope` → `warm` (передаём менеджеру в любом случае)

3. **Сгенерируй ответное письмо** (`reply_text`), соблюдая правила system
   prompt'а:
   - 2–3 коротких абзаца
   - ответить по существу на то, что спросил клиент
   - если уместно — предложить документ (прайс, каталог, листовку)
   - не повторяй подпись
   - не используй эмодзи, не кликбейт
   - для `reject` / `unsubscribe` — короткое вежливое подтверждение, без
     «может передумаете»
   - для `warm` — короткое «отлично, сейчас передаю контакт менеджера
     [AGENT_NAME_MANAGER], он свяжется сегодня-завтра и вышлет проект
     договора»
   - для `autoreply` — reply_text оставь пустой, `should_send = false`

4. **Subject** (`reply_subject`) — либо «Re: {исходная тема}», либо
   новая короткая тема если контекст сильно сменился.

5. **Вложения** (`attachments`) — имена файлов из каталога `documents/`.
   Разрешённые имена:
   - `pricelist_2026.pdf` — полный прайс-лист
   - `catalog_fitsiz_2026.pdf` — каталог 21 модели
   - `leaflet_element_classic.pdf` — листовка флагмана эконом (CLASSIC vs WELDER)
   - `leaflet_hd_ultra.pdf` — листовка флагмана проф (HD ULTRA vs FUBAG)
   - `commercial_offer_template.pdf` — только если запрошено КП
   - `certificate_tr_ts.pdf` — сертификат ТР ТС (только по запросу)

   НЕ прикладывай документы, которые уже были отправлены этому лиду (см.
   `sent_documents` выше). Если нужно отправить всё-таки повторно — напиши
   в `reply_text` объяснение.

6. **`is_warm`** — булево: `true` если это клиент, которому нужно передать
   менеджера (статус станет `warm`/`transferred`).

7. **`should_send`** — булево: `false` для автоответа или когда агент должен
   промолчать; `true` в остальных случаях.

  "new_status": "interested|negotiating|replied|warm|rejected|unsubscribed|...",
  "reply_subject": "строка темы ответа",
  "reply_text": "тело письма (2-3 абзаца)",
  "attachments": ["имя_файла.pdf"],
  "is_warm": false,
  "should_send": true,
  "reasoning": "1-2 предложения почему именно такая классификация"
}
